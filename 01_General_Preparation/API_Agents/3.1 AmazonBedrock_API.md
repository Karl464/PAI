# ☁️ Configuration & Connection Guide: Amazon Bedrock

This guide covers account creation, credential setup, model access configuration, and basic integration for **Amazon Bedrock** — following the same structure as the OpenAI and Azure guides.

> ⚠️ **Key difference from other providers:** Amazon Bedrock does NOT use API keys. Authentication is handled entirely through **AWS IAM credentials**. There is no `sk-` key to copy from a dashboard.

---

## 1️⃣ Create Your AWS Account & IAM User

1. Go to [aws.amazon.com](https://aws.amazon.com/) and create an account (or sign into an existing one)
2. In the [IAM Console](https://console.aws.amazon.com/iam/), create a **User** or use an existing one
3. Attach the **`AmazonBedrockFullAccess`** managed policy to the user
4. Under the user's **Security Credentials** tab, click **"Create access key"**
5. Choose **"Local code"** as the use case and copy both keys

Save credentials in your `.env` file:

```env
AWS_ACCESS_KEY_ID="AKIAxxxxxxxxxxxxxxxx"
AWS_SECRET_ACCESS_KEY="xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
AWS_DEFAULT_REGION="us-east-1"
BEDROCK_MODEL_ID="amazon.nova-lite-v1:0"
```

> 🔑 IAM access keys start with `AKIA`. Never commit them to version control.

**Alternative (recommended for production):** Use **AWS CLI** to configure credentials system-wide instead of `.env`:

```bash
pip install awscli
aws configure
# Prompts for: Access Key ID, Secret Access Key, Region, Output format
```

Once configured with `aws configure`, boto3 will pick up credentials automatically — no `.env` needed.

---

## 2️⃣ Enable Model Access in the Bedrock Console

> ⚠️ **Required step — models are NOT available by default.** You must manually request access for each model family you want to use.

1. Go to [Amazon Bedrock Console](https://console.aws.amazon.com/bedrock/)
2. In the left sidebar, click **"Model access"**
3. Click **"Modify model access"**
4. Check the model families you need (Amazon Nova, Anthropic Claude, Meta Llama, Mistral, etc.)
5. Click **"Request model access"** — approval is usually **instant** for most models

> 📌 Model access is **per-region**. If you switch AWS regions, you must re-enable access in that region.

---

## 3️⃣ Billing

Amazon Bedrock billing is fully integrated with your **AWS account** — no separate payment method required.

- Pay-as-you-go: billed per token, charged to your existing AWS invoice
- Track spending in [AWS Cost Explorer](https://console.aws.amazon.com/cost-management/)
- Set budget alerts in [AWS Budgets](https://console.aws.amazon.com/billing/home#/budgets) to avoid surprise costs
- **Free tier:** No Bedrock-specific free tier — every token is billed. New AWS accounts get a general $300 credit for 90 days.

## 5️⃣ Test Your Credentials

Create a simple script (`test_bedrock.py`) to confirm the connection works:

```python
import os
import boto3
from dotenv import load_dotenv

load_dotenv()

# boto3 reads AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_DEFAULT_REGION automatically
client = boto3.client("bedrock-runtime", region_name=os.getenv("AWS_DEFAULT_REGION", "us-east-1"))

response = client.converse(
    modelId=os.getenv("BEDROCK_MODEL_ID", "amazon.nova-lite-v1:0"),
    messages=[
        {"role": "user", "content": [{"text": "Write a short story about the Roman Empire."}]}
    ],
    inferenceConfig={"maxTokens": 200, "temperature": 0.7}
)

print(response["output"]["message"]["content"][0]["text"])
```

👉 If you see a short story printed, your setup is correct!

### A. Raw API Test Script (with error handling + token tracking)

```python
import os
import boto3
from botocore.exceptions import ClientError, NoCredentialsError
from dotenv import load_dotenv

load_dotenv()

client = boto3.client("bedrock-runtime", region_name=os.getenv("AWS_DEFAULT_REGION", "us-east-1"))

try:
    response = client.converse(
        modelId=os.getenv("BEDROCK_MODEL_ID", "amazon.nova-lite-v1:0"),
        system=[{"text": "You are a helpful assistant."}],
        messages=[
            {"role": "user", "content": [{"text": "Write a short story about the Roman Empire."}]}
        ],
        inferenceConfig={"maxTokens": 200, "temperature": 0.7}
    )
    print(f"✅ Connection Successful:\n{response['output']['message']['content'][0]['text']}")

    usage = response.get("usage", {})
    print(f"\n📊 Token Usage → Input: {usage.get('inputTokens')} | Output: {usage.get('outputTokens')}")

except ClientError as e:
    error_code = e.response["Error"]["Code"]
    print(f"❌ AWS Error [{error_code}]: {e.response['Error']['Message']}")
    if error_code == "AccessDeniedException":
        print("   → Check IAM permissions or that model access is enabled in the Bedrock console")
except NoCredentialsError:
    print("❌ No AWS credentials found. Run 'aws configure' or check your .env file.")
```

> **Note:** The `Converse` API is the recommended approach — it uses the same request/response format across all models, so switching from `amazon.nova-lite-v1:0` to `anthropic.claude-3-5-sonnet-20241022-v2:0` or `meta.llama3-70b-instruct-v1:0` requires only changing the `modelId`.

## 🔍 Tips & Best Practices

- **Never hardcode `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`** — use `aws configure`, IAM roles, or environment variables
- For production workloads running on AWS (EC2, Lambda, ECS), use **IAM roles** instead of access keys — no credentials in code at all
- Use the **`Converse` API** (not `InvokeModel`) for all chat tasks — it's model-agnostic and future-proof
- Enable model access **before** running code — `AccessDeniedException` with model errors is almost always a missing model access grant
- Set **AWS Budgets alerts** — Bedrock has no built-in spend cap; runaway agents can accumulate unexpected costs
- If you hit `ThrottlingException`, implement exponential backoff — free-tier accounts have conservative TPM limits
